A command to triage GlassFrog projects — six directions

Projects get captured well enough, but thin: no clear outcome, no definition of done, no next action. Modeled on the mattpocock triage, adapted to the fact that a Holacratic project is an outcome that needs a next-action and an owner — none of which GlassFrog requires. Six ranked directions, distinct in center of gravity, that compose into one layered system.

2026-07-20 topic: glassfrog-project-triage mode: repo-grounded focus: a project triage command

Grounding Context

The problem, precisely. GlassFrog captures projects with real ease — but the captured record is often under-specified: the description reads as an activity rather than an outcome, there's no stated definition of done, and no next physical action. The felt gap is that a project inbox fills with items nobody can act on without re-thinking them first.

  • There is already a triage twin for tensions. /holacracy:process-inbox walks unprocessed tensions per-role, presents a per-item triage block ([a] archive / [p] processed / [e] edit / [d] defer / [q] quit), and writes only on a per-item keystroke. A project triage command is the parallel surface for the project backlog — most of the architecture is already proven in commands/process-inbox.md.
  • The GlassFrog project write surface exists. The live MCP exposes glassfrog_list_role_projects / glassfrog_list_sub_projects / glassfrog_get_project, glassfrog_list_project_actions, and the writes glassfrog_update_project (description, status, value, effort, roi, private), glassfrog_create_action, glassfrog_create_role_project. A project can carry attached actions — so "next action" is a real, writable object, not a note.
  • But GlassFrog stores none of "well-formed" as a required field. Per glassfrog-api-constraints.md: project status is freeform text (no status machine); there is no definition-of-done field at all; and same-session list-back can be unreliable (treat creation-response IDs as the confirmation). Any notion of "done enough" has to be a rubric the command applies, plus a body/note convention for DoD — mirroring how tensions front-load their topic and use [GOVERNANCE]/[TACTICAL] body prefixes because there is no label field.
  • mattpocock triage, faithfully mapped. docs/agents/triage-labels.md already maps the mattpocock 5-state vocabulary (needs-triage, needs-info, ready-for-agent, ready-for-human, wontfix) to this repo. The transferable core is: assess each item, assign a state, and detect what's missing (needs-info). The Holacracy twist is that "needs-info" for a project is nearly always the same three gaps — outcome framing, next-action, owner — so the command can be far more specific than a generic tracker triage.
  • Strategy sets the rails. STRATEGY.md: "Develop, don't replace" — every interaction is scaffolding that builds practitioner competence; the AI drafts for human review and never auto-writes. And the key metric "tension capture→processing ratio" has an exact project analog: project capture→movement ratio — the share of captured projects that actually advance. A triage command is the operational lever on that metric.
  • Constitutional definition, load-bearing. A Project is "a specific outcome the role will work toward" (a desired end-state), distinct from the Next-Actions that move it. So the well-formedness test is constitutional, not stylistic: outcome-framed description + at least one concrete next-action + a clear owner role.

Topic Axes

A · Triage vocabulary

What states a project sorts into — well-formed, needs-outcome, needs-next-action, needs-owner, stale, blocked — and how faithfully it echoes the mattpocock 5-state set.

B · Well-formedness rubric

The diagnostic heart: what "done enough" means for a Holacratic project (outcome framing, next-action, owner, implicit DoD).

C · Route vs enrich

Does the command only label the gap, or draft the fix (reframed outcome, a next-action, a DoD line) for human confirmation?

D · Scope, selection & loop

Which projects (my roles / a circle / stale) and the walk shape — per-item, never-batched, quit-anytime, like process-inbox.

E · Write safety & API shape

update_project / create_action, draft-and-confirm, freeform-status & DoD-as-convention, same-session read caveat.

F · Alignment & authority

(added in follow-up) Well-placed, not just well-formed: does the project serve a goal, sit within the owner role's authority, and rest on the best-fit role/filler?

The Six Survivors

  • 1. Enrichment triage walker — thicken thin projects, draft-and-confirm
  • 2. Well-formedness rubric as a shared primitive
  • 3. Capture-time guardrail — stop thin projects at the source
  • 4. Next-action gate — the single highest-leverage check
  • 5. Faithful state-labeler — the minimal mattpocock clone
  • 6. Stalled-project sweep — guard the capture→movement ratio
These are layers, not either/or In your belt-and-suspenders-and-lifestyle-changes frame: #1 is the belt (fix the backlog in the moment), #6 is the suspenders (catch what silently stalls anyway), and #3 is the lifestyle change (stop under-specified projects being filed at all). #2 and #4 are the shared organs both depend on. The strongest scoping decision for ce-brainstorm may be "which layer ships first," not "which idea wins."
IDEA 1

Enrichment triage walker — thicken thin projects, draft-and-confirm

axes: B·C·D·E confidence: high complexity: medium-high
What
A /holacracy:triage-projects [circle] command that walks the actor's role projects and, for each, applies the well-formedness rubric, names the specific gap, and drafts the fix: a reframed outcome-shaped description, a proposed first next-action, and a DoD line. Per-project confirmation block offers [e] apply reframe (update_project), [a] add drafted next-action (create_action), [d] defer, [s] stale→archive, [q] quit. Nothing writes without a keystroke.
Why it fits
This is the only direction that touches the actual stated pain — "limited info, no DoD, no next action" — rather than just labeling it. It also lands squarely on the strategy: modeling good project form in the moment is the developmental scaffolding, and it moves the capture→movement ratio directly.
Basis
repo: commands/process-inbox.md (walk + per-item keystroke shape to clone) · skills/shared/tension-capture-flow.md (draft-and-confirm B-flow, never-batched, honest-error) · api: update_project + create_action exist · strategy: "Develop, don't replace" / capture→processing ratio.
Build sketch
New command + a shared skills/shared/project-well-formedness.md (Idea 2) it loads. Reuse the confirmation-block idiom verbatim. DoD lives as a DoD: convention in the description body (there's no field) — Idea 2 fixes the convention.
Risk / watch
Enrichment edges toward the AI manufacturing project meaning — the same line the constraints doc draws for AI-detected tensions. Mitigation: draft from the user's own words, confirm per-item, and treat the reframe as a proposal the human owns. Keep it a co-pilot, not an author.
IDEA 2

Well-formedness rubric as a shared primitive

axes: A·B·E confidence: high complexity: low
What
A single canonical reference — skills/shared/project-well-formedness.md — that defines the verdict every project surface uses: outcome-framed? (end-state, not activity) · has a defined next-action? · clear owner role? · implicit DoD legible? Plus the state vocabulary (well-formed / needs-outcome / needs-next-action / needs-owner / stale / blocked) and the DoD-as-body-convention decision.
Why it fits
Every other idea here needs this exact judgment. Extracting it once — the way tension-triage.md is loaded by five surfaces — means the command, the capture guardrail, and the sweep all agree on what "done enough" means. It's the cheapest, most reused thing you could ship, and it de-risks all the others.
Basis
repo: skills/shared/tension-triage.md (the shared-reference pattern, loaded by many surfaces) · constitution: project = outcome + next-actions · api: freeform status & no DoD field ⇒ rubric must live in the plugin, not GlassFrog.
Build sketch
Author the reference; give it a substitution-style test per gap ("if this were handed to another role-filler cold, could they act?"). Ship it as the foundation under Idea 1/3/4.
Risk / watch
Alone it changes nothing operationally — it's an enabling primitive, not a user-facing win. Rank it first to build, but it only pays off paired with a surface.
IDEA 3

Capture-time guardrail — stop thin projects at the source

axes: B·C·E confidence: medium-high complexity: medium
What
A /holacracy:capture-project create flow (twin of /holacracy:capture-tension) that won't file a project until it passes the rubric: it prompts for an outcome-shaped description and a first next-action, then fires create_role_project + create_action on confirmation. Under-specification is refused gently, with a one-line teaching of the missing element.
Why it fits
The root-cause / "lifestyle change" layer: triage debt exists because thin projects get filed in the first place. Fix the intake and the backlog stops filling with un-actionable items — the systemic move you'd pair with the in-the-moment fix.
Basis
repo: commands/capture-tension.md + skills/shared/tension-capture-flow.md (the exact create-flow template) · api: create_role_project, create_action · strategy: in-flow co-piloting at the point of work.
Build sketch
Mirror the capture-tension B-flow: resolve owner role → draft → per-item confirm → write. Loads Idea 2's rubric as the gate.
Risk / watch
Only governs projects created through the plugin — projects filed in the GlassFrog UI bypass it. So it needs #1 or #6 as the catch for everything born elsewhere. Also: don't let the gate become friction that pushes users back to the UI.
IDEA 4

Next-action gate — the single highest-leverage check

axes: B·C·D confidence: medium-high complexity: low
What
A deliberately narrow command that checks one thing: does each project have a defined next physical action (via glassfrog_list_project_actions)? For every project with none, surface it and offer to draft + file one next-action. Nothing else — no outcome reframe, no status debate.
Why it fits
In GTD-and-Holacracy orthodoxy a project without a next-action is stalled by definition. This is the cheapest slice that produces real movement, maps cleanly to the tactical-meeting "Projects" round, and is a natural MVP that Idea 1 can later subsume.
Basis
api: list_project_actions + create_action make the gap directly detectable and fixable · repo: Secretary/tactical surfaces already reason about projects-and-actions · strategy: capture→movement ratio.
Build sketch
List projects → for each, list actions → if empty, draft one → per-item confirm → create_action. Smallest viable command; ships in a fraction of Idea 1's surface area.
Risk / watch
A project can have a next-action and still be badly framed (activity-as-outcome). This catches stalls, not mis-framing — so it's a partial fix that leaves the "no DoD / no clear outcome" half of the pain on the table.
IDEA 5

Faithful state-labeler — the minimal mattpocock clone

axes: A·D confidence: high complexity: low
What
The most literal reading of "much like the mattpocock triage": walk the project backlog, assign each a state from a fixed vocabulary, and record it as a status: convention (freeform field) or a body prefix — [NEEDS-OUTCOME], [NEEDS-ACTION], [BLOCKED], [READY], [STALE]. No drafting of fixes; pure sorting so the backlog becomes scannable.
Why it fits
Closest to the reference you pointed at, lowest build cost, and it makes the backlog legible even before anyone fixes anything. A good staging post if you want the labeling discipline in place before investing in enrichment.
Basis
repo: docs/agents/triage-labels.md (the mattpocock 5-state map already lives here) · process-inbox.md (edit-body-to-prefix convention, exactly like tension [GOVERNANCE]/[TACTICAL]) · api: freeform status.
Build sketch
Walk + classify + write state to status/body via update_project. No create_action surface at all.
Risk / watch
Labeling without enrichment risks becoming a vanity ritual — a tidy backlog of items still nobody can act on. It sorts the pain; it doesn't resolve it. Ranked below the directions that actually move projects.
IDEA 6

Stalled-project sweep — guard the capture→movement ratio

axes: A·D·E confidence: medium complexity: medium
What
A back-of-loop sweep (twin of /holacracy:supersession-sweep) that surfaces projects showing no movement — old created_at, no recent action, stuck status — and offers to re-energize (draft a next-action) or archive (status: "Archived", the soft-collapse the constraints doc recommends over deletion). Optionally schedulable as an agentic routine.
Why it fits
It is the direct instrument on the strategy's project-movement metric and the "proactive practice-holding" track — it catches the tail that intake guardrails and in-the-moment triage both miss, and it keeps the backlog honest under stress.
Basis
repo: commands/supersession-sweep.md (sweep shape + session/recent scope) · skills/shared/agentic-routines.md (scheduling) · api: update_project(status), created_at available; archive-over-delete guidance.
Build sketch
List projects → heuristic for "stalled" (age + no actions + status) → per-item re-energize / archive / defer → write on keystroke.
Risk / watch
"Stalled" is a heuristic, and GlassFrog exposes no last-touched timestamp for the project itself — you infer staleness from created_at + action absence, which can false-positive on legitimately slow long-horizon projects. Confirm per-item; never auto-archive.

How they compose

Expansion — well-placed, not just well-formed

Follow-up round (2026-07-20). The original six ask "is this project a good statement of actionable work?" These four ask "is it the right project, on the right role, serving the right ends, within authority?" — a second quality family (axis F). The GlassFrog MCP already exposes the primitives each needs, so all four are feasible.

  • 7. Goal & target alignment — is this project serving any goal?
  • 8. Scope–authority validation — is this within the role's authority?
  • 9. Assignment fit — is this on the best-fit role/filler?
  • 10. Adversarial project-review panel — the synthesis that composes the lenses
Two families, one orchestration Ideas 1–6 are the well-formed family (actionable). Ideas 7–9 are the well-placed family (purposeful, authorized, well-owned). Idea 10 is the join: an adversarial review that runs each family's checks as an independent critic lens against a project and surfaces the holes. Whether 7–9 ship as standalone commands or as critics inside 10 is the central structural choice this expansion raises.
IDEA 7

Goal & target alignment — is this project serving any goal?

axes: F·C·E confidence: high complexity: low-medium
What
A check that, per project, asks whether it supports a goal or target the role/circle holds. Surfaces orphan projects (serving no stated goal) and, reciprocally, unbacked goals (goals with no supporting project). Offers to link a project to the goal it serves via glassfrog_link_goal_supporting_project.
Why it fits
Extends triage from "actionable" to "purposeful." A project that is perfectly well-formed but serves nothing the role is trying to achieve is still triage-worthy. And the reciprocal view (goals with no backing) is a strategy-health signal the plugin's governance-champion persona would want.
Basis
api: list_role_goals, list_goal_targets, list_goal_supporting_projects, and the direct write link_goal_supporting_project / unlink_goal_supporting_project all exist — this is the most directly-supported of the four · strategy: alignment to role/circle strategy.
Risk / watch
Not every legitimate project maps to a formal goal — some work is maintenance or tension-driven. Flag orphans as questions, not defects; don't imply every project must link to a goal.
IDEA 8

Scope–authority validation — is this within the role's authority?

axes: F·B confidence: medium-high complexity: medium
What
Checks a project's stated scope against the owning role's accountabilities and domains: is the role actually authorized to pursue this outcome? Flags projects that overreach the role (need a different role, a domain grant, or a governance change) or that reach into another role's domain.
Why it fits
Constitutionally, a project must fall within the role's purpose / accountabilities / domains — otherwise it's a governance question wearing a project's clothes. This is the check that turns a misfiled project into a sensed tension. It has a natural home next to your existing /holacracy:check-authority.
Basis
repo: commands/check-authority.md already reasons about role authority · api: get_role / glassfrog_get_role_context / list_role_domains return accountabilities + domains · constitution: project ⊆ role authority.
Risk / watch
"Within authority" is interpretive — accountabilities are prose. Advisory only: surface the concern and the governance route, never block or re-file. A false "overreach" flag on legitimate initiative-taking would be corrosive to trust.
IDEA 9

Assignment fit — is this on the best-fit role/filler?

axes: F confidence: medium complexity: medium
What
Asks whether the project sits on the best-fit owner, in two distinct readings it should keep separate: role-fit — given the scope, is this role the right one, or should another role own it? (governance-adjacent) — and person-fit — is the role's current filler resourced for it? (a Lead Link question). Surfaces the mismatch and the right forum; never reassigns.
Why it fits
Catches the "right work, wrong owner" failure the other checks miss. The person-fit reading connects directly to the Lead Link co-pilot's assignment work.
Basis
api: list_role_assignments, list_people, list_skills, role purposes available · repo: Lead Link skill reasons about role assignment · strategy: AI drafts / never auto-assigns.
Risk / watch
"Best fit" is the most judgment-heavy and the most politically sensitive of the four — highest risk of the AI manufacturing an opinion about people. Strictly advisory, and lean on person-fit only when the user asks; default to the structural role-fit reading.
IDEA 10

Adversarial project-review panel — the synthesis

axes: A·B·C·F confidence: high complexity: high
What
A /holacracy:review-project that runs a panel of independent adversarial critics against one project (or a backlog). Each critic gets a distinct lens — actionability (the rubric), goal-alignment, scope-authority, assignment-fit, and a red-team "what's missing / what could go wrong" critic whose only job is to poke holes. Findings are deduped, severity-ranked, and surfaced per-project for human confirmation; the human decides what to fix, and only then does anything write.
Why it fits
This is the direction you were reaching for — validate-scope and check-assignment "as critics in an agentic adversarial review," plus a dedicated hole-poker. It composes Ideas 2/7/8/9 as lenses rather than scattering them across commands, it matches the compound-engineering adversarial-verify pattern (independent critics beat one pass), and it is deeply developmental: showing a role-filler why their project is weak, from several angles, is exactly the "model good practice in the moment" mandate.
Basis
pattern: compound-engineering adversarial panel / judge-panel (independent lenses, dedup, severity filter) · repo: draft-and-confirm safeguard, develop-don't-replace · api: each critic reads its own MCP surface; writes gated per-finding.
Build sketch
Orchestrator dispatches one subagent per lens (parallel), each returns structured findings against the shared rubric (Idea 2); orchestrator dedups + ranks + presents; human confirms fixes that route to update_project / create_action / link_goal_supporting_project / a sensed tension. The narrow checks 7–9 can double as the standalone entry points and the panel's critics.
Risk / watch
Heaviest to build and the priciest to run (multi-agent). Over-critique is the real failure mode — a wall of nitpicks trains users to dismiss it. Needs a hard severity floor and a cap on findings per project. Keep every finding a question the human answers, never a verdict the AI enforces.

Critiqued & cut

The quality mechanism is explicit rejection, not optimistic ranking. What didn't survive, and why:

AFK / "ready-for-agent" split
Faithful to mattpocock's ready-for-agent vs ready-for-human, but it doesn't map: Holacratic projects aren't executed by AFK coding agents the way GitHub issues are. The valuable kernel — "is this well-formed?" — is already Idea 2. Cut as a distinct direction.
DoD-as-dedicated-field
Tempting, but GlassFrog has no definition-of-done field and no API to add one. Folded into Idea 2 as the "DoD-as-body-convention" decision rather than pursued as its own idea.
Developmental "teach-the-form" mode
Every triage verdict carrying a one-line teaching of why (activity vs outcome, etc.) is real value — but it's a cross-cutting quality layer, not a standalone command. Folded into Ideas 1, 3, and 5 as a property, per the strategy's develop-don't-replace mandate.
Tactical-meeting project-review prep
Genuinely useful, but it's an application of Ideas 2 + 4 into the existing Secretary / agentic-routines surface rather than a new triage direction. Noted as an adjacent packaging of Idea 4/6, not ranked separately.

Compound Engineering · Ideation

Should /holacracy:context load inherited context from enclosing circles?

Six ranked directions for closing the gap between the command's positional resolution today and the inherited Holacratic context (purpose, strategy, policies, domains) it never loads.

2026-06-16 topic: context-inherited-context mode: repo-grounded focus: inherited-context loading in /holacracy:context

Grounding Context

What the command does today. /holacracy:context resolves positional context only: actor identity (glassfrog_get_me), a flat role roster grouped by immediate circle (glassfrog_list_my_roles), and registered scheduled routines. It does no hierarchy traversal and never loads purpose, strategy, policies, or domains. The full resolution procedure is delegated to skills/shared/actor-and-role-resolution.md at an explicit "at most two tool calls" budget.

  • Recorded "keep it cheap" stance. ADR-0003 deliberately defers surfacing more in the command "to keep /holacracy:context cheap," filing the expansion as issue #17 (OPEN). Any default full-hierarchy load runs against this.
  • Feasibility contradiction (unresolved). The repo's glassfrog-api-constraints.md claims policies aren't readable via the API and there's no dedicated hierarchy endpoint. But the live GlassFrog MCP server connected to this session exposes glassfrog_get_role_context, glassfrog_list_role_policies, glassfrog_get_org_tree, and glassfrog_get_role_tree — and GlassFrog API v5 (May 2026) shipped a single-call role context bundle (identity, purpose, accountabilities, domains, policies, and "surrounding governance"), explicitly for AI agents. The repo's constraints doc appears stale relative to the live tool surface.
  • Existing load logic lives in skills. holacratic-ai-governance Pattern 1 already specifies "fetch the parent circle: strategy, policies, sister roles." Lead Link and Rep Link skills each load enclosing-circle content via glassfrog_get_circle independently. Convention: commands resolve/prime cheaply; skills load deep governance content.
  • Constitution v5 framing. Inherited context is an interpretive obligation (Art 4.2 — interpret governance in the context of the enclosing Circle and any Super-Circle) plus a prioritization duty (Art 2.3.3 — super-circle Strategies are mandatory inputs) up to the Anchor Circle. Purpose/Domains/Accountabilities do not structurally cascade — the cascade is interpretive context, not duplication.
  • Terminology mismatch. actor-and-role-resolution.md:73 narrates "Strategy on file says…", implying inherited strategy is loaded — but the command never loads it.

Ranked Ideas

  • 1. Ground-truth get_role_context first — the gating probe
  • 2. Argument-gate the deep load (--inherited); default stays cheap
  • 3. Surface binding constraints, not a governance dump (subsidiarity payload)
  • 4. Move the load into a shared reference; the command only primes
  • 5. Make loaded context reusable across skills (declaration block → ledger)
  • 6. User-asserted inheritance when GlassFrog is absent
1

Ground-truth get_role_context first — the gating probe

Axis B · TraversalConfidence 90%Complexity Low
Description
Before any design work, make one live glassfrog_get_role_context call against a real role and inspect the payload: does it return inherited "surrounding governance" (enclosing strategy + policies) in a single round-trip today? The answer partitions the entire design space — a working bundle makes "cheap" and "complete" the same thing; a thin bundle forces a per-level walk. Fold the traversal-mechanism question (does get_role_tree return ancestors or only descendants?) into the same probe.
Basis
direct: The repo's glassfrog-api-constraints.md states policies aren't exposed and there's no hierarchy endpoint, yet the live MCP server in this session lists glassfrog_get_role_context, glassfrog_list_role_policies, glassfrog_get_org_tree, and glassfrog_get_role_tree in its tool registry. external: GlassFrog v5 (May 2026) announced a single-call role-context bundle including "surrounding governance," explicitly for AI agents.
Rationale
Every other idea's cost calculus depends on this one fact. The verifier could not confirm these tools from repo files precisely because the repo's documentation predates them — that staleness is the gap. One probe replaces a stale doc with ground truth and chooses the architecture.
Downsides
Requires a live GlassFrog session with a populated org to be conclusive; a solo-operator org with a flat hierarchy may not exercise the "surrounding governance" field meaningfully.
SequencingDo this before committing to any of ideas 2–6. It is the cheapest high-leverage move in the set and it also lets you correct glassfrog-api-constraints.md as a side effect.
2

Argument-gate the deep load (--inherited); default stays cheap

Axis C · TriggerConfidence 85%Complexity Medium
Description
Add an explicit opt-in flag (/holacracy:context --inherited, optionally with a depth) that triggers enclosing-circle loading; absent the flag, the command holds its current two-call budget. Ship an ADR-0004 alongside that consciously names the cheap-default / opt-in-deep decision rather than silently eroding ADR-0003. The lightest viable default-payload variant — load only the immediate enclosing circle's Strategy (the one element Art 2.3.3 makes mandatory) — can live behind this flag rather than changing the default.
Basis
direct: Issue #17 (OPEN) explicitly proposes "Make it opt-in via an argument (/holacracy:context --with-tensions)"; ADR-0003 defers expansion "to keep /holacracy:context cheap." The flag pattern is already half-designed in the repo's own backlog.
Rationale
Honors the recorded constraint as a default while giving facilitators and Rep Links a deliberate escalation for governance prep. The matching ADR converts an informal deferral into an auditable decision — and becomes the repo's first compounded design record.
Downsides
Discoverability: an opt-in users don't know about delivers no value. Needs documentation in the command's argument-hint and README. Slight risk of flag proliferation if every command grows its own.
3

Surface binding constraints, not a governance dump (subsidiarity payload)

Axis A · PayloadConfidence 75%Complexity Medium
Description
Define the inherited payload as only what binds the leaf role: super-circle Strategies it must prioritize toward (Art 2.3.3), enclosing policies whose domain overlaps the role, and interpretation rulings touching its accountabilities (Art 4.2). Everything else above the role is loaded-but-suppressed. This is the CSS "computed value" move — resolve the cascade to what actually applies, don't reprint the whole ancestor ruleset.
Basis
external: Constitution v5 Art 4.2 (interpret in context of enclosing + Super-Circle) and Art 2.3.3 (super-circle Strategies mandatory), confirmed by independent web research; subsidiarity in federalism and CSS computed-style resolution as the structural analogies. reasoned: Token-cost research shows eager governance loads run 10k–60k tokens; a binding-only digest is a strict, much smaller subset that still satisfies the constitutional obligation.
Rationale
Gives a principled selection rule — "show what the Constitution makes binding on this role" — instead of "load everything up the chain." It is the difference between an output a Lead Link can act on and a wall of ancestor governance, and it answers the payload and cost axes with one rule.
Downsides
Requires judging domain overlap and ruling relevance — harder than a raw dump, and a wrong filter can hide a constraint that actually binds. Best paired with idea 1 confirming which elements are even readable.
4

Move the load into a shared reference; the command only primes

Axis D · PlacementConfidence 85%Complexity Medium
Description
Put the enclosing-chain load procedure where the repo already puts shared resolution logic — a new section in actor-and-role-resolution.md (or a sibling inherited-context-procedure.md) embedded by every role skill and reused by Pattern 1 — rather than re-implementing traversal in the command body. The command stays a thin pointer that primes and announces. This is also the cleanest fix for the line-73 mismatch: make "Strategy on file says…" true by giving the shared resolver an inherited-strategy step, so the narration matches behavior across all five skills at once.
Basis
direct: actor-and-role-resolution.md and authority-boundaries.md are already shared cross-skill via relative paths; commands/context.md:27 delegates its full procedure to the shared reference; Pattern 1 step 5 ("fetch the parent circle: strategy, policies, sister roles") is currently restated per skill. external: React context providers / DI scope resolution — one resolver, many callers.
Rationale
Single source of truth. Future v0.4 skills (policy-steward, circle-member) inherit the procedure for free, and the command-vs-skill convention the repo already follows is preserved rather than duplicated. Fixing the mismatch at the shared layer beats five per-skill patches.
Downsides
The shared reference grows in scope; needs care that the "command stays cheap" promise isn't quietly violated by skills that auto-run the deep load. Pairs with idea 2 to keep the trigger explicit.
Quick win inside this ideaThe narrowest standalone fix — rewrite actor-and-role-resolution.md:73 so the "Strategy on file says…" example is conditional ("if strategy was loaded: cite it; else: name the gap") — can ship immediately as a docs patch even before the larger relocation.
5

Make loaded context reusable across skills (declaration block → ledger)

Axis E · Cost & cachingConfidence 70%Complexity Medium
Description
Give the "don't re-fetch" convention an actual mechanism, on a spectrum: light — append a structured, visible block to the command output declaring what governance was loaded vs not, so skills read it from the transcript and skip duplicate fetches; heavy — write a session-scoped ledger (the materialized-view pattern) with a short TTL or event-based (governance-change) refresh that every skill reads. Pick the lightest version that removes the duplication for the common session.
Basis
direct: Rep Link's SKILL.md already instructs "it has already loaded governance context. Use what it loaded — don't re-fetch" — a convention with no mechanism; actor-and-role-resolution.md:32 notes the plugin does not persist resolved identity across sessions, so the transcript is the only cross-tier channel. external: database materialized views (pay the join once, accept bounded staleness); issue #17's own "cache … short TTL in a local ledger file."
Rationale
Reframes "keep cheap" from a reason-to-defer into a reason-to-cache: the cost concern in ADR-0003 is per-invocation; loading once and sharing makes the second-through-Nth use free across a multi-skill session.
Downsides
A ledger introduces staleness risk and a refresh-trigger to get right; the declaration-block variant is safe but only helps within one conversation. Governance-change detection needs a cheap signal.
6

User-asserted inheritance when GlassFrog is absent

Axis C · TriggerConfidence 70%Complexity Low
Description
Extend the existing disconnect fallback so the actor can declare inherited constraints — "my enclosing circle's strategy is X; these super-circle policies bind me" — and have those persist for the session alongside the asserted role. This is the robustness layer: it works when GlassFrog is disconnected and when the API simply can't return a piece (e.g., a policy the constraints doc says is unreadable).
Basis
direct: commands/context.md:20 already asks the user to declare role context when GlassFrog is off; glassfrog-api-constraints.md instructs "ask the user for policy context directly" when policies aren't available. reasoned: Whatever idea 1's probe finds, some inherited context (especially policy text) may always have to come from the user, so a declare-path is a required complement regardless of the API verdict.
Rationale
Belt-and-suspenders for inheritance: GlassFrog is the belt, user-assertion is the suspenders, and it makes the plugin usable for orgs running Holacracy without GlassFrog at all. It also keeps disconnect parity with how tactical.md already degrades.
Downsides
User-declared governance can be wrong or stale, and there's no audit trail back to GlassFrog; should be clearly marked as asserted-not-verified in any downstream use.
